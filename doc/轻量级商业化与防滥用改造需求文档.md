# 诛仙3副本战力计算器 - 轻量级商业化与防滥用改造需求文档

## 1. 项目背景与改造目标

**当前状态**：纯前端静态网页（部署于 Cloudflare Pages），核心数据（职业、副本技能等）分散在 `public/game_data/` 下的多个静态 JSON 文件中。

**改造目标**：
- 区分“免费版”和“付费版（全量版）”数据权限。
- 实现极简的“卡密（个人授权码）”验证机制，替代沉重的账号密码注册系统。
- **核心原则**：“防君子不防小人”，保持系统极度轻量化。
- 不引入数据库与复杂风控，优先实现 80% 功能；后续可扩展更强的防滥用能力。

### 核心优化思路
- **数据源维护**：保持原有的多个独立 JSON 文件（如 `classes.json`, `skills.json` 等）以便于人工维护和阅读。
- **数据隔离**：将 JSON 从 `public/` 移出，放入 Functions 可打包目录（如 `functions/data/`），随 Functions 编译发布，避免直接 URL 下载。
- **后端聚合**：后端 Function 直接使用编译期打包的 JSON（`import` 引入），执行过滤并合并为一个 JSON 对象返回前端，从而兼顾“维护成本”与“请求性能”。
- **Schema 标识**：在 JSON 数据中引入 `"isPremium": true` 标识，以便后端逻辑进行统一的“无感过滤”。

---

## 2. 技术栈与架构设计

充分利用现有的 Cloudflare 生态，实现零服务器成本的 Serverless 架构。
- **前端 (Frontend)**：React / Vite 框架，增加 LocalStorage 卡密缓存和激活 UI。
- **后端 (Backend)**：Cloudflare Pages Functions 提供轻量级 API 接口。
- **配置 (Config)**：环境变量维护静态白名单卡密（如 `VIP_CODES=code1,code2`）。
- **数据 (Data)**：JSON 随 Functions 打包发布，无需数据库。

---

## 3. 卡密策略（静态白名单）

- 使用环境变量 `VIP_CODES` 维护静态白名单，逗号分隔。
- 建议卡密格式保持一定复杂度（如 `zx-v-8d2a1f`），降低被猜测与爆破风险。
- 校验规则：卡密在白名单中则为 `premium`，否则返回 `free` 数据。
- 变更卡密需要重新部署（适合轻量项目与卡密量不大的阶段）。

### 可选扩展（未来）
- **签名卡密**：减少部署频率，卡密可离线生成（需新增签名校验）。
- **黑名单/KV**：对泄露卡密进行快速封禁（轻量新增能力）。

---

## 4. 后端 API 逻辑设计 (Cloudflare Pages Functions)

- **接口文件**: `functions/api/getData.ts`
- **请求格式**: `GET /api/getData`
  - **首选**：请求头 `X-VIP-Code: {卡密}`
  - **兼容**：`/api/getData?code={卡密}`

### 核心处理逻辑

1.  **读取卡密**：从 `X-VIP-Code` 或 query `code` 读取。
2.  **加载原始数据**：直接使用打包进 Functions 的 JSON（`import` 引入 classes, skills, dungeons, dungeons_monsters, buffs, rank_config）。
3.  **校验卡密**：
    - **无卡密或不在白名单**：返回过滤后的“免费版”数据。
    - **在白名单**：返回全量数据。
4.  **过滤逻辑**：
    - 遍历所有数据项，移除带有 `"isPremium": true` 的节点。
    - **引导性展示（可选）**：对付费职业/副本仅保留 `ID/Name` 等元信息，移除关键数值字段，便于前端显示灰态/锁图标并引导激活。
    - `dungeons` 被过滤后，对应的 `dungeons_monsters` 仅保留仍存在的副本（先得到合法 `DungeonID` 列表再过滤）。
5.  **版本字段**：返回数据中带 `version`（如 `20260314`），用于前端缓存失效与强制刷新。

**返回示例**：
- 免费：`{ status: 'free', version: '20260314', data: { classes: [...], skills: {...}, ... } }`
- 付费：`{ status: 'premium', version: '20260314', data: { ...全量数据... } }`

---

## 5. 前端改造需求

### UI 改造
- 在页面顶部增加一个“激活高级版”入口。
- 引导用户输入卡密，并在成功后显示“高级版已激活”勋章。

### 数据请求逻辑
- 废弃原有的 `DataService.ts` 中分散请求各 JSON 的逻辑。
- 统一调用 `/api/getData`，并将结果分发至 AppContext。
- **版本校验（可选）**：若前端未来对数据做缓存（如 LocalStorage），需对比 `version`，不一致则强制刷新。

### 本地存储 (LocalStorage)
- 自动保存已验证通过的卡密，并在提交前执行 `.trim()` 清洗。
- 验证失败不写入缓存，或提示用户重新输入。
- **容错回退（建议）**：`/api/getData` 失败时，回退到内置的最小默认数据，避免白屏。

---

## 6. 执行方案（轻量落地）

1. **数据标记与迁移**：给付费项加 `isPremium`，并将 JSON 从 `public/` 移至 `functions/data/`。确认构建产物 `dist` 中不再包含这些 JSON（检查 `vite.config.ts` 的 `publicDir` 配置）。
2. **后端接口**：新增 `functions/api/getData.ts`，实现白名单校验与 `isPremium` 过滤。
3. **前端接入**：`DataService` 统一请求 `/api/getData`，增加激活提示与 LocalStorage。
